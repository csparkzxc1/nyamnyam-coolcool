## 1. 태스크 번호 읽는 법

```
  T 3 0 1
  │ │ └─┴─ 태스크 순번 (01부터)
  │ └──── 스프린트 번호 (0~9)
  └────── Task의 T
```

**예시**
| 번호 | 해석 |
|---|---|
| **T001** | Sprint 0의 1번 태스크 (프로젝트 초기화) |
| **T101** | Sprint 1의 1번 태스크 (Supabase 연결) |
| **T301** | Sprint 3의 1번 태스크 (NextActionCard) |
| **T903** | Sprint 9의 3번 태스크 (앱스토어 제출) |

## 3. 태스크는 한 단위로 이렇게 구성됨

모든 태스크는 아래 7가지 정보를 담고 있습니다.

| 항목 | 뜻 |
|---|---|
| **제목** | 한 줄 요약 (예: "NextActionCard 컴포넌트") |
| **예상 N시간** | 개발자 한 명 기준 소요 시간 |
| **의존** | 이 태스크 전에 끝나 있어야 하는 다른 태스크 ID |
| **생성/수정 파일** | 어떤 파일을 만들거나 바꿀지 |
| **참조 문서** | PRD·스펙 문서에서 어느 부분을 봐야 하는지 |
| **구현 요점** | 어떻게 만드는지 핵심 지침 |
| **수락 조건** | "끝났다"라고 말할 수 있는 객관적 기준 |

**수락 조건 예시**: `npm test 통과`, `시뮬레이터에서 "Hello" 표시`, `타 유저가 접근 시 0 rows 반환`

## 4. 의존 관계 — 순서대로 해야 하는 이유

```
T001 (프로젝트 초기화)
  ↓ 프로젝트가 있어야
T002 (디자인 시스템) ─┐
T101 (Supabase 연결)─┤
                     ↓
                T102 (DB 스키마) ─┐
                T201 (인증) ──────┤
                                  ↓
                            T301 (홈 화면) ...
```

**원칙**: 같은 스프린트 안에서도 번호 순서가 곧 의존 순서입니다. T301을 건너뛰고 T306부터 시작하면 안 됩니다.

## 5. 자주 쓰는 용어

### 📱 앱 개발 용어

| 용어 | 뜻 |
|---|---|
| **MVP** | Minimum Viable Product. 최소한으로 쓸 만한 첫 버전 |
| **Phase 1** | 3개월 안에 만들 핵심 기능 묶음 |
| **스프린트 (Sprint)** | 관련 태스크 3~6개를 묶은 1~2주 작업 단위 |
| **컴포넌트 (Component)** | 화면을 이루는 부품 (버튼, 카드, 리스트 등) |
| **훅 (Hook)** | 컴포넌트가 쓰는 기능 조각 (`useSomething` 형태) |
| **마이그레이션 (Migration)** | DB 구조를 바꾸는 SQL 스크립트 |
| **RLS** | Row Level Security. "이 줄은 이 사람만 볼 수 있다" 규칙 |
| **OAuth** | 카카오·구글로 로그인하는 표준 방식 |
| **E2E 테스트** | 사용자가 실제로 쓰듯 자동으로 앱을 조작해보는 테스트 |
| **딥링크 (Deep Link)** | `nyamnyam://invite/abc` 같은 앱 전용 링크 |

### 🛠 우리 스택

| 이름 | 역할 |
|---|---|
| **Expo** | React Native 개발을 쉽게 해주는 도구 모음 |
| **React Native** | 코드 한 번으로 iOS·Android 앱 만드는 프레임워크 |
| **TypeScript** | JavaScript에 타입 안전성을 더한 언어 |
| **Supabase** | 데이터베이스·인증·실시간 동기화 통합 백엔드 |
| **NativeWind** | Tailwind CSS처럼 클래스로 스타일 쓰는 도구 |
| **Zustand** | 앱 전역 상태 저장소 (로그인 여부 등) |
| **TanStack Query** | 서버에서 받은 데이터 캐싱·갱신 |

## 6. Claude Code에 일 시키는 패턴

### 기본 사용 방법

```bash
# 한 개 태스크만
claude "IMPLEMENTATION_PLAN.md의 T001을 실행해줘"

# 여러 개 연속
claude "T001부터 T003까지 순서대로. 각 태스크 끝날 때마다 커밋해줘"

# 스프린트 단위
claude "Sprint 0 전체 완료해줘. 수락 조건 모두 통과하면 보고만 하고 멈춰"
```

### Claude Code에게 기대해도 되는 것 ✅

- 파일 생성·수정·삭제
- 터미널 명령 실행 (`npm install` 등)
- 테스트 실행 및 결과 확인
- 린트·타입 체크 통과시키기
- Git 커밋 메시지 작성
- 에러 메시지 보고 수정 반복

### Claude Code가 못 하거나 사람이 해줘야 하는 것 ⚠️

- Supabase 대시보드 가입 + 새 프로젝트 생성
- 카카오 개발자 센터 앱 등록
- 앱스토어 제출용 이미지·로고 디자인
- 실기기에서 알림·딥링크 실제 테스트
- 법무·개인정보 처리방침 검토
- 사업적 의사결정 (이름 확정, 가격 등)

## 7. 진행이 꼬였을 때 대처법

| 증상 | 해결 |
|---|---|
| Claude Code가 갑자기 엉뚱한 파일 만듦 | "CLAUDE.md의 §4 폴더 구조를 다시 보고 작업해" |
| 테스트 계속 실패한다고 함 | "수락 조건 하나씩 읽어주고 어디서 막혔는지 설명해" |
| 예상보다 오래 걸림 | "지금까지 뭐 했는지 요약하고 남은 건 뭔지 정리해" |
| 완료했다는데 실제로는 덜 됨 | 수락 조건 체크리스트 하나씩 직접 물어보기 |
| 태스크를 여러 개 건너뛰고 싶음 | 먼저 의존 관계 확인: "T501 의존 뭐야?" 물어보기 |

## 8. 오늘 바로 시작하려면

1. ✅ 5개 문서 모두 프로젝트 폴더에 배치
2. ✅ Node.js 20+, Git, Docker Desktop 설치 확인
3. ✅ Claude Code 설치 (`npm install -g @anthropic-ai/claude-code`)
4. ✅ 프로젝트 폴더에서 `claude` 실행
5. ▶️ 첫 명령: **"IMPLEMENTATION_PLAN.md의 T001을 실행해줘"**

끝나면 "완료 보고 받음" 확인하고 → "T002 이어서 진행해줘".
